```python
def send_visit_notification(reservation_id, visit_datetime):
    delay = (visit_datetime - datetime.now()).total_seconds()
    time.sleep(delay)  # 🔴 БЛОКИРУЕТ EVENT LOOP!
```

**Исправление:**
```python
async def send_visit_notification(reservation_id, visit_datetime):
    delay = (visit_datetime - datetime.now()).total_seconds()
```

2. **Потеря напоминаний при перезагрузке:**

**Решение:** Использовать Celery + Redis или APScheduler с персистентным хранилищем

## 5. АНАЛИЗ БЕЗОПАСНОСТИ

### 🔴 Критические уязвимости

1. **Hardcoded Credentials**
   - DB пароль в `database.py`
   - Telegram Bot токен в `telegram_bot.py`
   - Chat ID в `telegram_bot.py`

   **Риск:** Если код попадет в публичный репозиторий, все секреты скомпрометированы

2. **Слишком открытый CORS**
   ```python
   allow_methods=["*"],
   allow_headers=["*"],
   ```
   **Риск:** Возможны CSRF атаки

3. **Отсутствие аутентификации**
   - Любой может создать бронь
   - Нет проверки прав доступа

   **Риск:** Спам бронирования, фейковые брони

4. **Отсутствие rate limiting**
   **Риск:** DDoS атаки, создание тысяч бронирований

5. **API URL в коде frontend**
   **Риск:** Невозможно деплоить на production без изменения кода

### 🟡 Средние уязвимости

1. **Недостаточная валидация**
   - Нет проверки формата телефона
   - Нет проверки формата email
   - Можно создать бронь на прошедшую дату

2. **Отсутствие логирования**
   - Невозможно отследить инциденты
   - Нет audit trail

3. **SQL Injection**
   - ✅ Защищено через SQLAlchemy ORM

## 6. АНАЛИЗ ПРОИЗВОДИТЕЛЬНОСТИ

### Потенциальные проблемы

1. **Blocking Event Loop**
   ```python
   time.sleep(delay)  # Блокирует весь сервер!
   ```
   **Последствия:** При большом количестве бронирований API перестанет отвечать

2. **Нет индексов в БД**
   - Поиск по дате может быть медленным
   - Поиск по телефону неоптимален

3. **Нет кэширования**
   - Каждый запрос идет в БД
   - Нет кэша для частых запросов

4. **Нет пагинации**
   - Потенциально можно запросить все брони сразу
